### ASSISTANT
# Анализ промпта "Senior Software Architect"

## 📋 Что это за промпт?

Это **системный промпт** для настройки ИИ-ассистента в роли старшего программного архитектора. Он определяет роль, процессы, принципы и инструменты для проектирования масштабируемых систем.

---

## 🔢 Пошаговая структура промпта

### Шаг 1: Определение роли
```
Вы — старший архитектор ПО, специализирующийся на масштабируемом и поддерживаемом дизайне систем.
```
**Цель**: Задать контекст и экспертизу для всех последующих ответов.

---

### Шаг 2: Функциональные обязанности
| Обязанность | Описание |
|------------|----------|
| 🏗️ Проектирование архитектуры | Для новых функций и систем |
| ⚖️ Оценка компромиссов | Технических решений |
| 📐 Рекомендации паттернов | Best practices |
| 🔍 Выявление узких мест | Проблем масштабирования |
| 📈 Планирование роста | Будущая нагрузка |
| 🧩 Консистентность кода | Единые стандарты |

---

### Шаг 3: Процесс архитектурного ревью (4 этапа)

```
1️⃣ Анализ текущего состояния
   → Обзор существующей архитектуры
   → Документирование техдолга
   → Оценка ограничений масштабирования

2️⃣ Сбор требований
   → Функциональные и нефункциональные требования
   → Точки интеграции и потоки данных

3️⃣ Предложение дизайна
   → Диаграммы, компоненты, модели данных
   → API-контракты и паттерны интеграции

4️⃣ Анализ компромиссов
   → Плюсы/минусы/альтернативы/решение
```

---

### Шаг 4: Архитектурные принципы (5 столпов)

| Принцип | Ключевые аспекты |
|---------|-----------------|
| **Модульность** | SRP, слабая связность, чёткие интерфейсы |
| **Масштабируемость** | Горизонтальное масштабирование, stateless, кэширование |
| **Поддерживаемость** | Чистая организация, тестируемость, документация |
| **Безопасность** | Defense in depth, минимальные привилегии, валидация |
| **Производительность** | Эффективные алгоритмы, оптимизация запросов, lazy loading |

---

### Шаг 5: Библиотека паттернов

**Frontend**: Component Composition, Container/Presenter, Custom Hooks, Context, Code Splitting

**Backend**: Repository Pattern, Service Layer, Middleware, Event-Driven, CQRS

**Data**: Normalization/Denormalization, Event Sourcing, Caching, Eventual Consistency

---

### Шаг 6: Шаблоны документации

**ADR (Architecture Decision Record)** — структурированный формат для фиксации важных решений:
```markdown
# ADR-001: [Название]
## Context → ## Decision → ## Consequences → ## Status → ## Date
```

**Checklist проектирования** — чек-лист из 4 блоков:
- ✅ Функциональные требования
- ✅ Нефункциональные требования  
- ✅ Технический дизайн
- ✅ Операционная готовность

---

### Шаг 7: Антипаттерны (Red Flags)
⚠️ Big Ball of Mud, Golden Hammer, Premature Optimization, Not Invented Here, Analysis Paralysis, Magic, Tight Coupling, God Object

## 🎯 Для каких задач подходит этот промпт?

### ✅ Идеально подходит для:
| Задача | Пример |
|--------|--------|
| 🆕 Проектирование новой системы | "Спроектируй архитектуру микросервиса уведомлений" |
| 🔧 Рефакторинг существующей | "Как разбить монолит на модули?" |
| ⚖️ Выбор технологий | "REST vs GraphQL для мобильного бэкенда?" |
| 📊 Анализ масштабируемости | "Система не выдерживает нагрузку — что улучшить?" |
| 📝 Документирование решений | "Создай ADR для выбора базы данных" |
| 🔍 Архитектурный аудит | "Найди узкие места в текущей архитектуре" |
| 🧭 Онбординг команды | "Какие принципы архитектуры нам внедрить?" |

### ❌ Не подходит для:
- Написания конкретного кода функций (это задача разработчика)
- Отладки багов на уровне кода
- Дизайна UI/UX (визуального)
- Управления проектами и сроками

## 💡 Ключевые преимущества промпта

1. **Структурированность** — чёткий процесс от анализа до решения
2. **Баланс теории и практики** — принципы + конкретные паттерны
3. **Документирование** — встроенные шаблоны ADR и чек-листы
4. **Предотвращение ошибок** — список антипаттернов как "красные флаги"
5. **Гибкость** — подходит для разных стеков и масштабов

## 🚀 Как использовать максимально эффективно?

```
1. Начинайте с контекста: 
   "У нас есть [описание системы], нужно [цель]"

2. Указывайте ограничения:
   "Бюджет: 3 месяца, команда: 5 человек, стек: Node.js + PostgreSQL"

3. Запрашивайте формат:
   "Дай ответ в формате ADR" или "Предоставь чек-лист внедрения"

4. Уточняйте приоритеты:
   "Важнее скорость разработки или долгосрочная поддерживаемость?"
```
