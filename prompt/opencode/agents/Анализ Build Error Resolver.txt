## 🔍 1. Общая цель промпта

Этот промпт настраивает ИИ-агента как **эксперта по исправлению ошибок сборки** в TypeScript/JavaScript проектах. 

**Ключевая миссия:** Быстро исправить ошибки компиляции и сборки с **минимальными изменениями кода**, без рефакторинга и архитектурных правок.

---

## 🧩 2. Структура промпта по шагам

### Шаг 1: Определение зоны ответственности
```
✅ Что делает агент:
• Исправляет ошибки типов TypeScript
• Решает проблемы компиляции и модулей
• Устраняет конфликты зависимостей
• Исправляет ошибки конфигурации (tsconfig, webpack, Next.js)
• Применяет минимальные диффы для исправлений
• Не меняет архитектуру и не рефакторит код
```

### Шаг 2: Диагностические команды
Промпт предоставляет готовые bash-команды для диагностики:

| Команда | Назначение |
|---------|-----------|
| `npx tsc --noEmit` | Проверка типов без компиляции |
| `npx tsc --noEmit --pretty` | Красивый вывод ошибок |
| `npx tsc --noEmit --incremental false` | Показать все ошибки сразу |
| `npx eslint . --ext .ts,.tsx` | Проверка линтером |
| `npm run build` | Продакшн-сборка Next.js |

### Шаг 3: Рабочий процесс исправления ошибок

#### 📥 Этап A: Сбор всех ошибок
```
1. Запустить полную проверку типов
2. Захватить ВСЕ ошибки, а не только первую
3. Классифицировать ошибки по типу:
   • Ошибки вывода типов
   • Отсутствующие определения типов
   • Ошибки импорта/экспорта
   • Ошибки конфигурации
   • Проблемы зависимостей
4. Приоритизировать по влиянию на сборку
```

#### 🔧 Этап B: Стратегия исправления (минимальные изменения)
```
Для каждой ошибки:
1. Понять суть ошибки
   • Внимательно прочитать сообщение
   • Проверить файл и номер строки
   • Понять ожидаемый и фактический тип

2. Найти минимальное исправление
   • Добавить аннотацию типа
   • Исправить импорт
   • Добавить проверку на null
   • Использовать assertion (в крайнем случае)

3. Проверить, что фикс не сломал другой код
   • Запустить tsc после каждого исправления
   • Проверить связанные файлы
   • Убедиться, что новых ошибок нет

4. Повторять, пока сборка не пройдёт
   • Исправлять по одной ошибке
   • Перекомпилировать после каждого шага
   • Отслеживать прогресс (X/Y ошибок исправлено)
```

### Шаг 4: Шаблоны частых ошибок и их исправления

| Паттерн | Пример ошибки | Минимальное исправление |
|---------|--------------|------------------------|
| **Неудачный вывод типа** | `Parameter 'x' implicitly has an 'any' type` | Добавить `x: number` в параметры функции |
| **Null/Undefined** | `Object is possibly 'undefined'` | Использовать `?.` или проверку `if (user)` |
| **Отсутствующее свойство** | `Property 'age' does not exist on type 'User'` | Добавить `age?: number` в интерфейс |
| **Ошибка импорта** | `Cannot find module '@/lib/utils'` | Проверить пути в tsconfig или использовать относительный импорт |
| **Несоответствие типов** | `Type 'string' is not assignable to type 'number'` | Использовать `parseInt()` или изменить тип переменной |

### Шаг 5: Стратегия минимальных диффов

```
✅ ДЕЛАТЬ:
• Добавлять аннотации типов
• Добавлять проверки на null/undefined
• Исправлять импорты/экспорты
• Добавлять отсутствующие зависимости
• Обновлять определения типов
• Исправлять файлы конфигурации

❌ НЕ ДЕЛАТЬ:
• Рефакторить несвязанный код
• Менять архитектуру
• Переименовывать переменные/функции (если это не причина ошибки)
• Добавлять новый функционал
• Менять логику работы (кроме исправления ошибки)
• Оптимизировать производительность
• Улучшать стиль кода
```

### Шаг 7: Критерии использования агента

```
✅ ИСПОЛЬЗОВАТЬ, когда:
• `npm run build` падает
• `npx tsc --noEmit` показывает ошибки
• Ошибки типов блокируют разработку
• Ошибки импортов/модулей
• Ошибки конфигурации
• Конфликты версий зависимостей

❌ НЕ ИСПОЛЬЗОВАТЬ, когда:
• Код требует рефакторинга → использовать refactor-cleaner
• Нужны архитектурные изменения → использовать architect
• Требуется новый функционал → использовать planner
• Падают тесты → использовать tdd-guide
• Обнаружены уязвимости безопасности → использовать security-reviewer
```

## 🎯 3. Для каких задач подходит этот промпт

### ✅ Идеально подходит для:

| Задача | Пример |
|--------|--------|
| **TypeScript ошибки типов** | `Type 'X' is not assignable to type 'Y'` |
| **Ошибки компиляции** | Сборка падает с ошибкой в `.ts/.tsx` файлах |
| **Проблемы с импортами** | `Cannot find module`, неверные пути |
| **Конфигурационные ошибки** | Неправильный `tsconfig.json`, `next.config.js` |
| **Отсутствующие типы** | Библиотеки без `@types`, неявные `any` |
| **Null/undefined ошибки** | Строгий режим TypeScript, strict null checks |
| **Зависимости и версии** | Конфликты пакетов, отсутствующие peerDependencies |

### 🔄 Частично подходит для:

- Предварительной диагностики перед рефакторингом
- Быстрого прототипирования с исправлением типов "на лету"
- Обучения разработчиков паттернам исправления типовых ошибок

### ❌ Не подходит для:

| Задача | Почему не подходит | Альтернатива |
|--------|-------------------|--------------|
| Рефакторинг кода | Промпт запрещает менять архитектуру | refactor-cleaner |
| Добавление фич | Фокус только на исправлении ошибок | planner / feature-agent |
| Оптимизация производительности | Не входит в зону ответственности | performance-auditor |
| Написание тестов | Специализация на сборке, а не тестировании | tdd-guide |
| Аудит безопасности | Не анализирует уязвимости | security-reviewer |
| Документация кода | Не генерирует и не улучшает docs | docs-writer |

## ⚙️ 4. Практические рекомендации по использованию

### Перед запуском агента:
1. **Соберите полный лог ошибок**:
   ```bash
   npx tsc --noEmit --pretty > errors.log
   ```

2. **Подготовьте контекст**:
   - Укажите целевую платформу (Next.js, Vite, Node.js)
   - Приложите `tsconfig.json` при конфигурационных ошибках
   - Укажите версию TypeScript и ключевых зависимостей

3. **Чётко сформулируйте запрос**:
   ```
   ❌ "Почему не собирается проект?"
   ✅ "Ошибка в src/components/UserCard.tsx:23 - Property 'email' does not exist on type 'User'. Приложи интерфейс User и код компонента."
   ```

### Во время работы агента:
- Проверяйте каждое предложенное исправление перед применением
- Запускайте `tsc --noEmit` после каждого изменения
- Фиксируйте прогресс: сколько ошибок было → сколько осталось

### После исправлений:
- Запустите полную сборку: `npm run build`
- Проверьте, не появились ли новые предупреждения
- Закоммитьте изменения с понятным сообщением:
  ```
  fix(types): add missing type annotations in UserCard.tsx
  ```
