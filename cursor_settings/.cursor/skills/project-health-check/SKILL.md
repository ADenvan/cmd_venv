# Project health check

## Назначение

Этот навык помогает быстро оценить базовое состояние текущего репозитория: структуру проекта, управление зависимостями и наличие/запуск базовых проверок качества (линтеры и тесты).

Используйте этот навык, когда:
- пользователь просит «health check» проекта;
- нужно понять, насколько репозиторий готов к развитию или передаче другим разработчикам;
- требуется краткий обзор сильных сторон и основных рисков по проекту.

## Quick start

Выполняя проверку, придерживайтесь следующего сценария.

### 1. Определение типа проекта

- Найдите ключевые файлы, указывающие на стек:
  - Python: `pyproject.toml`, `requirements.txt`, `requirements-dev.txt`, `Pipfile`;
  - JavaScript/TypeScript: `package.json`, `pnpm-lock.yaml`, `yarn.lock`;
  - другие экосистемы — по аналогичным конфигурационным файлам.
- Обратите внимание на наличие lock‑файлов (`poetry.lock`, `requirements-lock.txt`, `package-lock.json`, `pnpm-lock.yaml` и т.п.).

### 2. Структура проекта

- Оцените, насколько структура понятна:
  - есть ли отдельная папка для исходников (например, `src/`, `app/`, `backend/`, `frontend/`);
  - есть ли отдельная папка для тестов (`tests/`, `test/`, `__tests__/`);
  - присутствуют ли конфиги для инструментария (`.cursor/`, `.github/`, конфиги линтеров и форматтеров).

### 3. Зависимости

- Проверьте:
  - есть ли явный файл зависимостей (например, `pyproject.toml`, `requirements.txt`, `package.json`);
  - есть ли lock‑файл, фиксирующий версии;
  - нет ли явных признаков «захламлённости» зависимостей (много пакетов при маленьком проекте, дублирующие инструменты и т.п.) — на уровне мягких рекомендаций.
- Не обновляйте зависимости автоматически — только анализ и рекомендации.

### 4. Базовые проверки (тесты)

- Найдите стандартные команды запуска тестов:
  - для Python — `pytest`, `python -m unittest` или команды в `pyproject.toml`/Makefile;
  - для JS/TS — `npm test`, `yarn test`, `pnpm test` или скрипты в `package.json`.
- Если команда тестов очевидна и безопасна:
  - при запросе пользователя запустите её;
  - зафиксируйте в отчёте, что тесты запускаются, и их результат (проходят/падают, кратко — что сломано).

### 5. Линтеры и форматтеры

- Определите, какие инструменты настроены, по конфигурационным файлам:
  - Python: `ruff`, `flake8`, `pylint`, `black`, `isort` и их конфиги;
  - JS/TS: `eslint`, `prettier` и их конфиги;
  - другие линтеры — по аналогии.
- Если команда запуска линтера/форматтера очевидна и пользователь не против:
  - запустите её;
  - кратко опишите в отчёте наличие/отсутствие ошибок или объём проблем.

## Checklist

Используйте этот чек‑лист во время проверки.

- [ ] **Структура проекта понятна**: выделены каталоги для исходников и тестов.
- [ ] **Файл зависимостей присутствует**: есть основной файл (`pyproject.toml`, `requirements.txt`, `package.json` и т.п.).
- [ ] **Есть lock‑файл** или понятно, почему он не используется.
- [ ] **Команда тестов определена** и при запросе пользователя может быть запущена.
- [ ] **Линтер/форматтер настроен** (есть конфиги) и при необходимости может быть запущен.
- [ ] **Нет критичных сбоев**: тесты и базовые команды запускаются, либо проблемы явно зафиксированы в отчёте.

## Report format

Оформляйте результат проверки по следующему шаблону.

```markdown
## Project health summary

**Общий вывод:** краткое резюме (например: «состояние хорошее, есть несколько улучшений по структуре»).

### Сильные стороны
- Пункт 1
- Пункт 2

### Проблемы и риски
- [Критично/Важно/Минорно] Краткое описание проблемы.
- [Критично/Важно/Минорно] Краткое описание проблемы.

### Рекомендованные следующие шаги
1. Самый приоритетный шаг.
2. Следующий важный шаг.
3. Дополнительные улучшения (по желанию).
```

Адаптируйте формулировки под конкретный проект, но сохраняйте общую структуру: резюме → сильные стороны → риски → шаги.

## When not to use

Не используйте этот навык, когда:

- требуется **глубокий аудит безопасности** (поиск уязвимостей, анализ прав доступа, соответствие стандартам);
- нужна **оценка производительности** (нагрузочное тестирование, профилирование, сложная оптимизация);
- проверяется **инфраструктура и CI/CD** (пайплайны, деплой, мониторинг).

В этих случаях:
- выполняйте целевой анализ (без опоры на этот чек‑лист), либо
- предлагайте пользователю провести отдельные специализированные проверки.
